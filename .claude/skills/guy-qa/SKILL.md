# Guy QA Skill — מתודולוגיית בדיקת איכות

מדריך גיא לביצוע QA עקבי ואפקטיבי.

---

## 5 ממדי הצ'קליסט — הגדרות מפורטות

### 1. רלוונטיות לבריף
**מה בודקים:** האם התוכן עונה על מה שמיכל ביקשה?

✅ **עובר כאשר:**
- הנושא המרכזי תואם את הבקשה המקורית
- כל הנקודות המרכזיות מהבריף מכוסות
- אין חריגה משמעותית לנושאים זרים

❌ **נכשל כאשר:**
- חלקים גדולים מהתוכן לא קשורים לבריף
- חסרות נקודות מפתח שנדרשו
- הכיוון הכללי שגוי

---

### 2. התאמת תמונות (N/A אם אין תמונות)
**מה בודקים:** האם כל תמונה מחזקת את הטקסט שלידה?

✅ **עובר כאשר:**
- התמונה ממוקמת ליד הטקסט הרלוונטי
- התמונה מוסיפה ערך (לא רק קישוט)
- ה-path בקובץ תקין וה-file קיים ב-`outputs/`

❌ **נכשל כאשר:**
- התמונה לא קשורה לטקסט שלידה
- ה-path שגוי / הקובץ לא קיים
- תמונה חסרה שצוינה כנדרשת בבריף

---

### 3. סגנון כתיבה
**מה בודקים:** האם הטון, הקצב והמבנה תואמים את דוגמאות הסגנון?

✅ **עובר כאשר:**
- משפטים קצרים ל-בינוניים (כמו בדוגמאות)
- פנייה ישירה ב"את" (אם הדוגמאות כך)
- טון רגשי-מקצועי — לא רשמי מדי, לא שטחי
- מבנה ברור: הוק → גוף → CTA

❌ **נכשל כאשר:**
- משפטים ארוכים מדי / שפה אקדמית
- פנייה שגויה (אתה/הם במקום את)
- חסר הוק בפתיחה
- CTA חסר או חלש

---

### 4. שלמות
**מה בודקים:** האם כל האלמנטים הנדרשים קיימים?

רשימת חובה:
- [ ] כותרת ראשית
- [ ] פסקת פתיחה / הוק
- [ ] גוף מחולק לסעיפים / שקפים
- [ ] סיכום / CTA
- [ ] מקור (אם התוכן נחקר מהרשת — חובה)

✅ **עובר כאשר:** כל האלמנטים הנדרשים קיימים
❌ **נכשל כאשר:** אחד או יותר חסרים

---

### 5. פורמט
**מה בודקים:** האם הפורמט הטכני תקין?

✅ **עובר כאשר:**
- Markdown תקין (headers, bold, רשימות)
- אם יש שקפים — ממוספרים ועקביים
- אין טקסט שנשבר, תגיות פתוחות, שגיאות עריכה
- שם הקובץ תקין (`Output/<filename>`)

❌ **נכשל כאשר:**
- Markdown שבור
- שקפים ללא מספור / מספור לא עקבי
- שגיאות הדפסה גסות בשמות / כותרות

## כתיבת הערות תיקון אפקטיביות

### כלל הזהב: ספציפי + ניתן לביצוע

| ❌ לא כך | ✅ כך |
|---------|-------|
| "הסגנון לא טוב" | "פסקה 3: המשפטים ארוכים מדי. פצלי כל משפט ל-2" |
| "חסר משהו" | "חסר CTA בסוף — הוסיפי שאלה לקוראים כמו בדוגמאות" |
| "התמונה לא מתאימה" | "התמונה בשקף 4 מציגה X אבל הטקסט עוסק ב-Y" |

### מבנה הערה טובה:
```
[מיקום מדויק]: [מה הבעיה] → [מה צריך לעשות]
```

---

## Anti-Patterns — מה גיא לא עושה

| ❌ אסור | הסבר |
|---------|-------|
| לשכתב קטעים | זה תפקיד יעל, לא גיא |
| לפסול ללא סיבה | כל ❌ חייב הסבר מפורש |
| לבקר סגנון אישי | אם הסגנון תואם את הדוגמאות — זה מאושר |
| לדרוש שינויים שמעבר לבריף | גיא בודק מה ביקשו, לא מה הוא היה רוצה |
| להשאיר דוח ריק | גם אישור מלא מקבל דוח עם כל הסעיפים ✅ |

## פורמט שם קובץ הדוח

```
QA_Reports/YYYYMMDD_<slug>_approved.md      ← אישור
QA_Reports/YYYYMMDD_<slug>_revision1.md     ← תיקון סיבוב 1
QA_Reports/YYYYMMDD_<slug>_revision2.md     ← תיקון סיבוב 2
QA_Reports/YYYYMMDD_<slug>_escalated.md     ← סיבוב 3 → מיכל
```

`<slug>` = שם הקובץ מ-Output/ ללא סיומת
